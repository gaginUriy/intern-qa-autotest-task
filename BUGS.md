### BUG-011
| Параметр | Значение |
|---|---|
| **ID** | BUG-011 |
| **Заголовок** | Неверные HTTP-коды ответа при операции удаления ресурса |
| **Описание** | При удалении существующего ресурса возвращается статус `200` вместо ожидаемого `204 No Content`. При повторном удалении уже удалённого ресурса возвращается `400 Bad Request` вместо `404 Not Found`, что нарушает REST-конвенции. |
| **Предусловия** | В системе создан тестовый ресурс с известным `ID`. |
| **Шаги для воспроизведения** | 1. Создать ресурс методом `POST /api/1/item` и сохранить его `ID`.<br>2. Отправить `DELETE /api/2/item/:id` для первого удаления.<br>3. Отправить `DELETE /api/2/item/:id` повторно. |
| **Фактический результат** | Первый запрос возвращает `200 OK`, второй запрос возвращает `400 Bad Request`. |
| **Ожидаемый результат** | Первый запрос возвращает `204 No Content`. Повторный запрос возвращает `404 Not Found`. |

### 🔍 Уточнения и заметки для команды
1. **Типы данных и валидация:** Используется целочисленный тип (`integer`), однако поля не должны принимать отрицательные значения. Требуется уточнение требований: добавить ли проверку `>= 0` на уровне API или изменить тип данных в БД.
2. **Структура ответа статистики:** При запросе статистики по одной карточке товара возвращается массив данных. Необходимо уточнить контракт API: ожидается ли получение объекта с единственной записью или массив допустим по архитектуре.
3. **Метод DELETE:** В техническом задании отсутствует описание метода `DELETE`. Требуется согласовать логику удаления, ожидаемые коды ответов (`204`/`200`/`404`) и поведение при повторных запросах к уже удалённому ресурсу.
4. **Счётчик просмотров:** Уточнить бизнес-логику инкрементации счётчика: должен ли он увеличиваться при каждом `GET`-запросе на получение карточки, или требуется внедрение защиты от накрутки (например, учёт сессии/IP или задержка повторного зачёта).
5. **Карточки «плюшевый» и «пятнистый котёнок»:** Проверить на соответствие логике фильтрации и корректность отображения атрибутов.
6. **Количество результатов выдачи:** Возникают сомнения в корректности количества возвращаемых БД карточек (результатов мало). Требуется проверить ответ API и выполнить прямой запрос к БД для верификации.
7. **Рекомендации:** Выяснить причину появления большого количества «пятнистых котов» в блоке рекомендаций. Проверить алгоритм персонализации и весовые коэффициенты тегов.
8. **Фильтры совместимости:** В фильтрах «Ладит с детьми» и «Ладит с другими животными» отсутствует опция «Нет», при этом присутствует кнопка «Да». Необходимо свериться с техническими требованиями и уточнить дизайн/логику состояния фильтров.
